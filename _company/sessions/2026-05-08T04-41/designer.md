# 📐 UX/UI 디자인 전략 브리프: 데이터 중심의 경험 설계 원칙 (v1.0)

본 문서는 단순한 '미적 트렌드'가 아닌, **복잡하고 중요한 데이터를 다루는 B2B(기업 간 거래) 환경**에 최적화된 사용자 경험(UX) 및 사용자 인터페이스(UI)의 구조화 원칙을 제시합니다. 특히, AI 기반 시뮬레이션 결과와 같이 신뢰성과 명확성이 가장 중요한 정보를 전달하는 데 중점을 둡니다.

---

## Ⅰ. 핵심 UX 트렌드: 정보의 '소유권' 이동 (Shift of Ownership)

최신 디자인은 사용자에게 *정보를 소비*하게 하는 것이 아니라, **사용자가 데이터를 직접 조작하고 의미를 발견**하게 만드는 방향으로 진화했습니다. 모든 결과물(Web/App/Proposal)에 이 개념이 적용되어야 합니다.

| 트렌드 영역 | 키워드 및 정의 | Web/App 적용 방식 (UX Flow) | 제안서/문서 적용 방식 (Deliverable) |
| :--- | :--- | :--- | :--- |
| **Agentic UX** | 사용자의 요청 하나로 여러 AI 에이전트가 단계적으로 작업하고, 그 *과정*을 시각화하여 보여주는 경험. | 단일 버튼 클릭 → '분석 중...' (에이전트 A 실행) → '법규 검토 중...' (에이전트 B 실행) → 최종 통합 보고서 제시. **(프로세스 투명성)** | 제안 목차를 순차적 워크플로우 다이어그램으로 구성하고, 각 단계별 분석 툴킷을 포함하여 고객이 직접 변수를 돌려보게 한다. |
| **Interactive Data Visualization** | 정적인 그래프 대신, 사용자가 슬라이더나 필터 조작만으로 데이터의 가정이 바뀌고 결과가 즉시 시뮬레이션되는 인터랙티브 경험. | 핵심 KPI(예: ROI)를 메인 대시보드에 배치하고, 그 주변 변수들(CAPEX, 위험 계수 등)을 노출된 컨트롤 패널로 제공한다. **(직접적 조작감)** | 최종 보고서의 핵심 페이지를 PDF가 아닌, 웹 기반 '실시간 시뮬레이터 툴킷'으로 대체하여 배포한다. |
| **Micro-Interaction & Feedback Loop** | 사용자가 특정 액션을 취할 때마다 (버튼 클릭, 필드 입력 등) 미묘하고 즉각적인 피드백(색상 변화, 애니메이션 등)을 제공하여 신뢰도를 높이는 것. | 데이터 로딩 시 뼈대가 되는 스켈레톤 UI를 사용하여 '무엇이 언제 올지' 예측하게 하고, 에러 발생 시 친절하고 구체적인 복구 가이드와 함께 실패 원인을 제시한다. | 제안서 내의 CTA(Call-to-Action) 버튼을 단순한 링크가 아닌, "다음 단계 진행"이라는 의미를 담은 동적 요소로 설계한다. |

## Ⅱ. 정보 구조화 원칙 (Information Architecture Principles)

복잡하고 재정적인 내용을 다루기 때문에, 인지 부하(Cognitive Load) 최소화가 최우선 목표입니다. 정보를 '덩어리'로 나누고, 가장 중요한 것부터 보여줘야 합니다.

1.  **점진적 공개 (Progressive Disclosure):**
    *   **원칙:** 한 화면에 모든 정보를 넣으려 하지 마십시오. 사용자가 필요할 때만 깊은 정보를 단계적으로 펼쳐서(Toggle) 보여줍니다.
    *   **실행 예시:** '상세 분석 보기' 버튼을 통해 관련 법규나 통계 데이터를 숨겨두고, 기본 화면에는 최종 KPI와 결론만을 배치합니다. (최소한의 정보로 최대의 영향력을 발휘).

2.  **정보 계층 구조화 (Hierarchical Flow):**
    *   **원칙:** 모든 요소는 명확하게 '가장 중요 → 중간 중요 → 세부 설명' 순서로 위계가 정해져야 합니다.
    *   **실행 예시:** 페이지의 가장 상단 10%에 **"핵심 결론(Executive Summary)"**을 배치합니다. (예: "본 솔루션은 평균적으로 X년 Y개월 내 ROI를 확보하여 Z억 원의 재무적 리스크 헤지 효과를 가져옵니다.")

3.  **인지 지도 및 온보딩 (Mental Model & Onboarding):**
    *   **원칙:** 사용자가 우리 솔루션을 어떻게 이해해야 할지, 마치 가이드가 안내하듯이 명시적으로 돕습니다. 처음 접하는 사람을 위해 '이 섹션은 무엇을 하는 곳입니다'라는 설명을 의무화합니다.
    *   **실행 예시:** 시뮬레이터의 첫 실행 화면에 "여기서 CAPEX를 입력하면, 시간 흐름에 따른 기대 ROI가 계산됩니다."와 같은 명확한 튜토리얼 플로우를 제공해야 합니다.

## Ⅲ. 필수 기술 및 접근성 가이드라인 (Technical & Accessibility Standards)

디자인의 완성도는 *누구나* 사용할 수 있어야 하며, 개발 단계부터 이 기준을 반영해야 합니다.

### 1. WCAG 준수 (웹 콘텐츠 접근성 지침)
| 항목 | 기준/요구사항 | 디자인 적용 필터 | 중요도 |
| :--- | :--- | :--- | :--- |
| **색상 대비** | 최소 AA 레벨(WCAG 2.1: 4.5:1 이상). 특히 텍스트와 배경 간의 명암비가 절대적으로 확보되어야 함. | 주요 강조 색상만 사용하고, 본문 텍스트는 높은 명도/채도의 조합을 피한다. (진한 네이비/검정 계열) | **필수** |
| **포커스 관리** | 키보드 사용자(마우스 미사용자)가 모든 인터랙티브 요소(버튼, 필터 등)에 초점을 맞췄을 때 시각적으로 명확하게 포커스링이 표시되어야 한다. | `outline: 2px solid #007bff;`와 같은 구체적인 스타일 코드를 적용하여 개발 단계에서부터 테스트해야 함. | **필수** |
| **스크린 리더 지원** | 모든 이미지, 아이콘, 복잡한 그래프는 대체 텍스트(`alt tag`)를 포함하거나, 논리적 순서(Semantic HTML)를 가지도록 구조화되어야 한다. | 단순히 '분석'이라고 적지 않고, "2026년 대비 예상 손실액 비교 차트"와 같이 구체적인 설명을 제공한다. | **필수** |

### 2. 개발 방법론 관점의 UI 고려사항
*   **반응형 설계 (Responsive First):** 데스크탑 화면을 기준으로 디자인하지 말고, 가장 작은 모바일 환경에서도 핵심 KPI가 가려지지 않고 한눈에 들어오도록 레이아웃부터 구상해야 합니다.
*   **컴포넌트 기반 사고:** 모든 UI 요소(버튼, 카드, 슬라이더 등)는 재사용 가능한 독립적인 컴포넌트로 설계하고 관리합니다. 이는 향후 템플릿 표준화 작업의 핵심 기반이 됩니다.
