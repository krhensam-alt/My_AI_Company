## Ⅰ. 개발 방법론 및 운영 철학 (Agile & DevOps Principles)

| 영역 | 핵심 개념 | 적용 방안 (ROI 시뮬레이터 기준) | ROI/위험 관리 관점의 이점 |
| :--- | :--- | :--- | :--- |
| **애자일 (Agile)** | 반복적 개발 및 사용자 피드백 기반 개선. | **MVP → PoC → 전사 도입 로드맵**을 스프린트 단위로 분할. 매 스프린트마다 '핵심 가치 증명(PoC)'에 집중하고, 고객/이해관계자에게 즉시 시연하여 검증받는다. | 불필요한 기능 개발 비용 지출 최소화 (Cost Overrun 방지). 시장 변화에 대한 민첩성 확보. |
| **데브옵스 (DevOps)** | 개발(Dev)과 운영(Ops)의 통합 자동화 및 지속적인 배포/개선. | 1. **CI (Continuous Integration):** 모든 코드는 Git을 통해 버전 관리하고, Pull Request 발생 시 자동 테스트 실행 의무화. 2. **CD (Continuous Delivery):** 결과물은 Streamlit 또는 웹 서비스 형태로 빌드되어 자동으로 스테이징 환경에 배포되도록 파이프라인 구축. | 시스템 장애 위험 최소화(Risk Mitigation). 예측 불가능한 운영 고장 대비 보험금 역할을 수행. |
| **테스트 전략** | 단위 테스트, 통합 테스트, 시나리오 기반 테스트. | 핵심 로직 (손실액 계산, ROI 산출)은 반드시 Mocking Data를 사용한 **단위/통합 테스트 코드로 작성하고 커버리지를 90% 이상 유지**한다. 실제 외부 API 호출은 최종 배포 직전 단계로 제한한다. | '기술적 부채(Technical Debt)' 발생을 사전에 막아 장기적인 운영 비용 절감 효과를 창출. |

## Ⅱ. 기술 아키텍처 패턴 선택 (Scalability & Resilience)

ROI 시뮬레이터의 구조는 '프론트엔드/백엔드'의 역할을 명확히 분리하여, 한 부분이 실패해도 전체 시스템이 무너지지 않도록 설계해야 합니다.

| 아키텍처 패턴 | 정의 및 역할 | ROI 시뮬레이터에 적용 (개발 단계) | 장점 및 고려사항 |
| :--- | :--- | :--- | :--- |
| **모놀리식 (Monolithic)** | 모든 기능을 하나의 애플리케이션으로 통합. (초기 MVP 적합) | 🟢 초기 Streamlit PoC 개발에 사용. UI와 계산 로직을 한 파일에서 처리하며, 빠른 검증에 최적화된다. | **장점:** 구현 속도가 빠름. **단점:** 기능 확장에 따른 장애 전파 위험(Failure Domain)이 높다. (초기 목표 달성 후 분리 필요) |
| **마이크로 서비스 (Microservices)** | 애플리케이션을 독립적인 작고 책임 구역별 서비스 단위로 분할. (장기 목표) | 🟡 계산 엔진(Calculation Service), 데이터 전처리(Data Ingestion Service), API 게이트웨이 등으로 분리한다. | **장점:** 각 서비스가 독립적이므로, 한 곳의 장애가 전체 시스템을 마비시키지 않는다. (운영 안정성 보험 극대화). **단점:** 서비스 간 통신 복잡도 및 관리가 어렵다. |
| **서버리스 (Serverless)** | 인프라 관리 없이 사용한 만큼만 비용을 지불하는 클라우드 컴퓨팅 모델 (AWS Lambda, Azure Functions 등). | 🔴 **데이터 처리/API 백엔드 로직**에 적용한다. 특정 이벤트(예: 데이터 파일 업로드)가 발생할 때만 코드를 실행하여 비용 효율성을 극대화한다. | **장점:** 운영 인력 투입 최소화 (TCO 절감). 사용량 기반 과금으로 예측 가능한 예산 책정 가능. |

## Ⅲ. 필수 언어 및 스택 조합 (Tech Stack Selection)

| 기술 요소 | 추천 스택/언어 | 핵심 역할 및 이유 | 장점 (ROI 관점) |
| :--- | :--- | :--- | :--- |
| **백엔드 / 로직** | Python (Pandas, NumPy) | **데이터 처리, 통계 분석, 복잡한 시뮬레이션 계산.** 데이터 과학 분야의 표준 언어이며, 라이브러리 생태계가 매우 풍부하여 '분석적 깊이'를 확보하는 데 가장 유리하다. | 빠른 개발 속도와 강력한 데이터 연산 능력으로, MVP 구현 및 기능 확장이 용이함. |
| **프론트엔드 / UI** | Streamlit (MVP) → React/Vue + Flask/FastAPI (확장) | 초기에는 Python 기반의 Streamlit을 사용하여 로직과 UI를 빠르게 결합한다. 이후 복잡도가 높아지면, 백엔드는 FastAPI로 구축하고 프론트는 React로 분리하여 전문적인 인터랙티브 경험(Interactive Data Viz)을 제공한다. | **가성비 최고의 접근:** 낮은 초기 비용으로 높은 수준의 사용자 인터페이스 구현이 가능하며, 확장 시 업계 표준 스택으로 전환할 수 있다. |
| **데이터베이스** | PostgreSQL (PostGIS 등) | 구조화된 데이터 저장 및 관리. 특히 '산업별 규제 준수 실패 법적 패널티'와 같은 복잡한 지리/시간 데이터를 다룰 때 강력하다. | 높은 트랜잭션 안정성과 확장성이 보장되어, 대규모 운영 데이터를 처리할 수 있다. |
